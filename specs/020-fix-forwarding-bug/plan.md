# Implementation Plan: Fix Message Forwarding Bug

**Branch**: `020-fix-forwarding-bug` | **Date**: 2026-02-22 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/020-fix-forwarding-bug/spec.md`

## Summary

Fix a critical bug where messages from newly-added source channels are not forwarded. The GramJS `NewMessage` event handler is registered at startup with an immutable `chats` filter — channels joined later are never picked up. The fix removes the static `chats` filter (which is client-side only, per research) and adds dynamic channel tracking via callbacks from `ChannelOpsConsumer` to `ListenerService`.

## Implementation Details

### Change 1: ListenerService — Remove static filter, add dynamic tracking

**File**: `apps/worker/src/listener/listener.service.ts`

1. In `start()`, change event handler registration from:
   ```typescript
   new NewMessage({ chats: channelIds, incoming: true })
   ```
   to:
   ```typescript
   new NewMessage({ incoming: true })
   ```

2. Add public methods:
   ```typescript
   addChannel(telegramId: number): void {
     this.activeChannelIds.add(telegramId);
     this.logger.info({ telegramId }, 'channel_added_to_listener');
   }

   removeChannel(telegramId: number): void {
     this.activeChannelIds.delete(telegramId);
     this.logger.info({ telegramId }, 'channel_removed_from_listener');
   }
   ```

### Change 2: ChannelOpsConsumer — Add notification callbacks

**File**: `apps/worker/src/listener/channel-ops-consumer.ts`

1. Add optional callbacks to constructor:
   ```typescript
   constructor(
     private readonly channelManager: ChannelManager,
     logger: pino.Logger,
     private readonly onChannelJoined?: (telegramId: number) => void,
     private readonly onChannelLeft?: (telegramId: number) => void,
   )
   ```

2. In `processJob()`, invoke callbacks after successful operations:
   - After `joinChannel()` succeeds: `this.onChannelJoined?.(result.telegramId)`
   - After `leaveChannel()` succeeds: `this.onChannelLeft?.(job.data.telegramId!)`

### Change 3: main.ts — Wire callbacks

**File**: `apps/worker/src/main.ts`

Pass listener methods as callbacks when creating `ChannelOpsConsumer`:
```typescript
const channelOpsConsumer = new ChannelOpsConsumer(
  channelManager,
  logger,
  (telegramId) => listener.addChannel(telegramId),
  (telegramId) => listener.removeChannel(telegramId),
);
```

### Change 4: Tests

**File**: `apps/worker/test/listener.spec.ts`
- Test that `addChannel` adds to `activeChannelIds`
- Test that `removeChannel` removes from `activeChannelIds`
- Test that messages from dynamically-added channels are processed
- Test that messages from removed channels are ignored
- Test that event handler is registered without `chats` filter

**File**: `apps/worker/test/channel-ops-consumer.spec.ts`
- Test that `onChannelJoined` callback is invoked after successful join
- Test that `onChannelLeft` callback is invoked after successful leave
- Test that callbacks are not invoked on failed operations
- Test backward compatibility (callbacks are optional)
